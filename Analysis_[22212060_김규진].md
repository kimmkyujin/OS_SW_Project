## 1. Introduction
본 문서는 자취생 요리 보조 어플리케이션인 든든(Deun Deun)의 분석 결과에 대한 내용을 담고 있다. <br/>
기존의 레시피 어플들이 표준적인 조리 환경만을 가정하는 한계를 보완하고자, 사용자의 현재 상황에 맞춘 유동적인 레시피 변환 어플리케이션을 설계하였다.

1) Prominent Features
- 본 어플리케이션은 단순히 레시피를 나열하는 것이 아닌, 사용자가 현재 보유한 조리도구와 재료 상황에 따라 레시피를 동적으로 변경하여 최적의 조리법을 제공한다.
- 요리 초보자가 겪는 재료 낭비와 조리 실패 상황을 줄여주며, 1인 가구의 건강한 식생활 자립을 돕는 것을 핵심 가치로 한다.
- 사용자가 레시피를 스크랩하여 관리하고, 기간별 인기 레시피를 추천받는 기능을 통해 지속적인 요리 활동을 지원한다.

2) Goals
- 본 Analysis 단계에서는 User, Administrator, Server, System 간의 상호작용 시나리오를 Use case 분석을 통해 명확히 정의한다.
- System이 User의 요청에 따라 올바른 응답을 반환하도록 하는 기능 로직과 예외 처리 흐름을 구체화한다.
- Domain Analysis를 통해 시스템을 구성하는 핵심 클래스와 클래스 간의 관계를 명확히 하여 이후 설계 단계의 기반을 마련한다.
- User Interface Prototype을 통해 사용자 관점에서의 화면 흐름과 인터랙션을 시각적으로 표현한다.

| 4 | 중복이 아니면 회원정보를 데이터베이스에 저장 후 로그인 화면(Use case #1)을 띄운다. |

#### Use case #6: Storage & Care Guide

**GENERAL CHARACTERISTICS**

| **Field** | **Description** |
| :---: | --- |
| **Summary** | 레시피에 사용된 식재료의 보관 방법과 조리도구의 올바른 관리 및 세척법을 제공하는 기능 |
| **Scope** | Deun Deun |
| **Level** | User level |
| **Status** | Analysis |
| **Primary Actor** | User |
| **Preconditions** | 사용자가 레시피 상세 페이지(Use case #4)를 보고 있어야 한다. |

| **Success Post Condition** | 해당 레시피에서 사용된 식재료 보관법 및 도구 관리 정보가 화면에 나타난다. |
| **Failed Post Condition** | 사용자는 보관 및 관리 방법에 대한 정보를 보지 못하고 상세 레시피 화면(Use case #4)을 유지한다. |

**MAIN SUCCESS SCENARIO**

| **Step** | **Action** |

## 3. Domain Analysis

**1) User**
사용자 클래스. 회원가입 및 로그인 정보를 담고 있으며, 일반 사용자와 관리자 권한을 구분하여 시스템 접근 범위를 결정하는 클래스이다.

**2) Recipe**
레시피 클래스. 요리명, 기본 인분 수, 난이도, 상세 조리 순서 등 요리의 핵심 데이터를 포함하며 시스템의 가장 중심이 되는 정보를 관리하는 클래스이다.

**3) Ingredient**
식재료 클래스. 레시피에 필요한 재료 정보를 담고 있다. 각 재료의 중요도(필수/선택), 표준 단위(g, 큰술 등) 및 신선도 유지를 위한 보관법 정보를 포함한다.

**4) CookingTool**
요리도구 클래스. 조리에 필요한 도구 정보를 관리한다. 해당 도구가 없을 때 사용할 수 있는 대체 도구 정보와 조리 후 도구별 올바른 세척 및 관리법을 포함한다.

**5) RecipeAdapter**
레시피 변환 클래스. 사용자가 선택한 인원수, 보유 재료, 조리 도구 옵션에 따라 기존 레시피의 수치 및 조리법을 변경하여 맞춤형 레시피를 생성하는 기능을 담당한다.

**6) Scrap**
스크랩 클래스. 사용자가 나중에 다시 보고 싶은 레시피를 저장하는 기능을 수행한다. 어떤 사용자가 어떤 레시피를 저장했는지에 대한 정보를 관리한다.

**7) Inquiry**
문의사항 클래스. 서비스 이용 중 발생하는 버그 제보나 레시피 요청 내역을 저장한다. 문의 유형, 내용 및 관리자의 답변 상태와 답변 내용을 포함한다.

**8) Recommendation**
추천 클래스. 시스템 내의 조회수와 스크랩 수 데이터를 분석하여 주간, 월간, 연간 단위로 사용자에게 인기 있는 레시피 TOP 5 목록을 산출하는 로직을 관리한다.

* **로그인 버튼:** 입력한 정보로 데이터베이스 인증을 시도한다. 미입력 상태이거나 정보가 일치하지 않으면 팝업을 띄우고 로그인 화면을 유지한다. (Use case #1 예외 시나리오 처리)
* **회원가입 링크:** 하단 텍스트를 누르면 회원가입 화면으로 이동한다. (Use case #2 Trigger)

<br/>

**■ 화면 설명**
신규 사용자가 계정을 생성하는 화면이다. (Use case #2)

* **가입 버튼:** 입력한 데이터로 계정 생성을 요청한다. 시스템은 빈칸 확인 및 데이터 중복 검사(아이디, 이메일, 전화번호)를 수행한다.
* **예외 처리:** 누락된 항목이 있거나 중복된 정보가 발견되면 팝업을 띄우고 현재 화면을 유지한다. (Use case #2 예외 시나리오 처리)
* **완료:** 정상적으로 가입이 완료되면 데이터베이스에 정보를 저장하고 로그인 화면으로 이동한다.

### 4.8. Inquiry Screen

## 5. Glossary

본 분석 문서 및 '든든' 시스템에서 사용되는 주요 용어에 대한 정의이다.

| **용어** | **설명** |
| --- | --- |
| **든든** | 본 프로젝트에서 개발하는 1인 가구 및 자취생을 위한 맞춤형 요리 보조 어플리케이션의 명칭 |
| **레시피 변환** | 사용자가 선택한 인원수, 현재 보유하고 있는 식재료 및 조리 도구 상황에 맞추어 기존 레시피의 계량 수치와 조리법을 동적으로 변경하여 제공하는 본 시스템의 핵심 기능 |
| **스크랩** | 사용자가 마음에 드는 레시피를 나중에 다시 열람하기 위해 본인의 계정에 저장하는 기능 |
| **맞춤 옵션** | 원래 레시피에 요구되는 특정 식재료나 조리 도구를 사용자가 보유하고 있지 않을 때, 시스템이 조리 로직에 맞게 제안하거나 사용자가 선택할 수 있는 다른 식재료 및 도구 |
| **팝업** | 시스템이 사용자에게 경고, 오류 메시지, 작업 완료 등의 중요한 정보를 전달하기 위해 현재 화면 위에 일시적으로 띄우는 작은 창 |
