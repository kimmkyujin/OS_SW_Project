## 1. Introduction

본 문서는 Analysis 단계 문서에 이어지는 든든(Deun Deun) 어플리케이션의 Design 단계 문서이다.

### 1.1 Summary
든든(Deun Deun)은 자취생과 1인 가구를 위한 맞춤형 요리 보조 어플리케이션으로, 사용자의 보유 재료 및 조리도구 상황에 맞게 레시피를 동적으로 변환하여 제공하는 것을 핵심 기능으로 한다. Analysis 단계에서 정의된 10개의 Use Case를 바탕으로, 본 Design 문서에서는 시스템을 구성하는 클래스 구조, 주요 기능 흐름(Sequence Diagram), 상태 전이(State Machine Diagram)를 구체적으로 정의한다.


### 1.2 Important Points of Design

- **레시피 변환 중심 설계:** 사용자의 인원수·재료·조리도구 선택에 따라 레시피를 동적으로 변환하는 것이 든든 앱의 핵심 기능이며, 모든 다이어그램에서 이 흐름을 중심으로 설계한다.

- **권한 기반 접근 제어:** 일반 사용자(USER)와 관리자(ADMIN)의 역할을 명확히 구분하여, 각 기능에 대한 접근 범위를 제한한다.

- **데이터 무결성 유지:** 클래스 간 다대다 관계를 연관 클래스로 분리하고, 원본 데이터를 직접 수정하지 않는 방향으로 설계한다.

- **예외 처리 흐름 명확화:** 네트워크 오류, 빈 입력, 중복 데이터 등 Use Case에서 정의된 예외 시나리오를 각 다이어그램에서 구체적으로 표현한다.

## 3. Sequence diagram

### 3.1 Login
<img width="566" height="577" alt="image" src="https://github.com/user-attachments/assets/55824a66-9635-4be2-afdd-500b753490bf" />
<br/>

본 다이어그램은 사용자가 로그인을 시도할  때의 Sequence diagram 입니다. 전체 흐름은 클라이언트의 1차 입력값 검증과 서버 및 데이터베이스의 2차 인증 단계로 나뉩니다.

사용자가 아이디와 비밀번호를 입력하고 로그인 버튼을 누르면, App은 자체적으로 입력값 공백 여부를 검사합니다. 이때 첫 번째 alt 분기문이 작동하여 입력값이 누락된 경우 서버 요청 없이 즉시 "아이디와 비밀번호를 모두 입력해주세요"라는 팝업을 띄우고 흐름을 종료합니다. 반면 값이 모두 채워진 상태라면 App은 Server에 login(loginId, password) API 요청을 보냅니다.

요청을 받은 Server는 유효성 검증을 위해 DB에 회원정보 조회를 요청하고 결과를 반환받습니다. 이 결과에 따라 두 번째 alt 분기문이 실행되는데, 일치하는 정보가 없다면 Server는 인증 실패 신호를 보내 App이 "아이디 또는 비밀번호가 틀렸습니다"라는 경고 팝업을 띄우게 만듭니다. 반대로 일치하는 정보가 존재하면 Server는 권한 정보를 포함한 인증 성공(role 포함) 신호를 보내며, 이를 받은 App이 메인 화면으로 이동(replace)하면서 전체 로그인 프로세스가 완료됩니다.

### 3.2 Register
<img width="432" height="794" alt="image" src="https://github.com/user-attachments/assets/b410c117-2fc6-4506-ac41-3bfb893f28d4" />
<br/>

본 다이어그램은 사용자가 회원가입을 시도할  때의 Sequence diagram 입니다. 전체 흐름은 클라이언트 측의 필수 입력값 검증과 서버 측의 단계별 중복 검사(아이디, 이메일, 전화번호), 그리고 최종 저장 단계로 진행됩니다.

사용자가 로그인 화면에서 회원가입 버튼을 누르면 App은 회원가입 화면을 표시합니다. 유저 정보를 입력한 후 가입 버튼을 클릭하면 App은 필수 입력값의 공백 여부를 먼저 검사합니다. 첫 번째 alt 분기문에서 입력값이 누락되었다면 서버 요청 없이 즉시 "모든 정보를 입력해주세요"라는 팝업을 띄우고 흐름을 종료합니다. 모든 값이 존재하면 App은 Server에 register(Info) API 요청을 보냅니다.

요청을 받은 Server는 가장 먼저 DB에 아이디 중복 검사를 요청합니다. 여기서 아이디가 중복되면 Server는 중복 오류를 반환하고 App은 "이미 사용 중인 아이디입니다"라는 팝업을 출력합니다. 중복이 없다면 이어서 이메일 중복 검사를 진행하며, 이메일마저 중복될 경우 "이미 사용 중인 이메일입니다"라는 팝업을 띄우게 됩니다. 이메일 검사를 통과하면 마지막으로 전화번호 중복 검사를 수행하여, 중복 시 "이미 사용 중인 전화번호입니다"라는 팝업으로 사용자에게 피드백을 줍니다. 모든 중복 검사에서 이상이 없는 중복 없음 상태가 되면, Server는 DB에 회원정보 저장을 요청하고 저장이 완료되는 순간 App에 회원가입 성공 신호를 보냅니다. 마지막으로 이를 받은 App이 로그인 화면으로 이동하면서 전체 회원가입 프로세스가 완료됩니다.

본 다이어그램은 사용자가 키워드를 통해 레시피를 검색할 때의 Sequence diagram 입니다. 전체 흐름은 클라이언트 측의 검색어 유효성 검증과 서버 및 데이터베이스를 통한 레시피 조회, 그리고 결과에 따른 화면 전환 단계로 진행됩니다.

사용자가 검색창에 키워드를 입력하고 검색 버튼을 누르면, App은 가장 먼저 검색어의 공백 여부를 검사합니다. 이때 첫 번째 alt 분기문이 작동하여 검색어가 없는 경우 서버 요청 없이 즉시 "검색어를 입력해주세요"라는 메시지를 표시하고 흐름을 종료합니다. 반면 검색어가 존재하는 상태라면 App은 Server에 해당 키워드를 전달합니다.

키워드를 받은 Server는 DB에 키워드 기반 Recipe 조회를 요청하고 그 결과를 반환받습니다. 이 조회 결과에 따라 두 번째 alt 분기문이 실행됩니다. 만약 일치하는 데이터가 없는 일치하는 레시피 없음 상태라면 Server는 빈 결과를 반환하고, App은 "해당 레시피가 없습니다"라는 메시지를 화면에 표시합니다. 반대로 일치하는 데이터가 존재하는 상태라면 Server는 레시피 목록을 반환하며, App은 이를 받아 화면에 레시피 목록을 출력합니다. 마지막으로 사용자가 목록에서 특정 레시피를 클릭하면 상세 화면으로 이동하면서 전체 검색 프로세스가 완료됩니다.

### 3.4 View Recipe
<img width="882" height="693" alt="image" src="https://github.com/user-attachments/assets/7c66705f-dd9b-4b1d-8e31-0216e851c2e8" />

본 다이어그램은 사용자가 특정 레시피의 상세 정보를 요청할 때의 Sequence diagram 입니다. 전체 흐름은 레시피의 다각적인 데이터 조회와 조회수 증가 처리, 그리고 네트워크 상태에 따른 분기 대응 단계로 진행됩니다.

사용자가 레시피 목록에서 특정 레시피를 클릭하면, App은 Server에 getDetails(recipeId) API 요청을 보냅니다. 요청을 받은 Server는 해당 레시피의 구체적인 정보들을 가져오기 위해 DB에서 Recipe, RecipeStep, RecipeIngredient, RecipeCookingTool 테이블을 통합적으로 조회한 뒤 상세 데이터를 반환받습니다. 이후 통신 상태에 따라 alt 분기문이 실행됩니다.

만약 네트워크 오류나 데이터 로드 실패가 발생한 상태라면, Server는 오류 응답을 보내고 App은 사용자에게 "오류!" 알림창을 표시한 뒤 이전 목록 화면을 그대로 유지합니다. 반면 정상 응답 상태일 경우, Server는 DB에 incrementViewCount(recipeId)를 요청하여 해당 레시피의 조회수를 1 증가시킵니다. 카운트 처리가 완료되면 Server는 App에 최종 레시피 상세 데이터를 반환합니다. App은 이를 바탕으로 별 버튼과 옵션 버튼이 포함된 상세 조리법 화면을 사용자에게 띄워주며, 이후 사용자가 뒤로가기를 실행하면 이전 화면으로 안전하게 되돌아가면서 프로세스가 마무리됩니다.

### 3.7 Recommend Recipe

### 3.8 View Scrap
<img width="596" height="882" alt="image" src="https://github.com/user-attachments/assets/a1facd12-4204-49c4-bbc0-e1c913c307ad" />
<br/>

본 다이어그램은 사용자가 레시피를 스크랩하고, 보관소에서 스크랩 목록을 조회하거나 취소할 때의 Sequence diagram 입니다. 전체 흐름은 스크랩 등록, 목록 조회 및 상태별 분기, 그리고 스크랩 취소(opt) 단계로 구성됩니다.

사용자가 레시피 상세 화면에서 하트 아이콘을 클릭하면, App은 Server에 Scrap.add(userId, recipeId) 요청을 보냅니다. Server는 DB에 스크랩 데이터를 저장한 후 완료 신호를 반환하며, App은 하트 아이콘을 활성화하여 사용자에게 등록 완료 피드백을 줍니다. 이후 사용자가 하단 탭의 '보관소' 버튼을 누르면 App은 Server를 통해 DB의 Scrap과 Recipe 테이블을 JOIN 조회하여 사용자의 스크랩 목록을 요청합니다. 이때 alt 분기문이 실행됩니다.

첫 번째 분기인 스크랩 목록 없음 상태일 경우, 빈 목록이 반환되어 App은 화면에 "스크랩된 레시피가 없습니다!"라는 안내 문구를 표시합니다. 두 번째 분기인 스크랩 목록 존재 상태일 경우, 레시피 목록(썸네일, 요리명)이 반환되어 App 화면에 출력됩니다. 이 목록에서 특정 레시피를 클릭하면 상세 화면으로 이동합니다. 마지막으로 사용자가 상세 화면에서 하트 아이콘을 다시 클릭하는 스크랩 취소(opt) 상황이 발생하면, App은 Server에 Scrap.remove 요청을 보냅니다. Server가 DB에서 해당 데이터를 삭제하고 취소 완료 신호를 보내면, App은 하트 아이콘을 비활성화하고 보관소 목록에서 해당 레시피를 제거하며 프로세스가 완료됩니다.

### 3.9 Send Inquiry
<img width="712" height="781" alt="image" src="https://github.com/user-attachments/assets/bde49358-cb7c-46b3-b2df-cf020774e52f" />
<br/>

본 다이어그램은 사용자가 애플리케이션 내에서 문의사항을 작성하고 전송할 때 User, App, Server, DB 간에 이루어지는 데이터 흐름과 조건별 예외 처리 과정을 나타냅니다. 전체 흐름은 입력값 유효성 검증, 전송 결과에 따른 분기 대응, 그리고 작성 중 취소(opt) 상황에 대한 UI 제어 단계로 진행됩니다.

사용자가 하단 탭의 '문의사항' 버튼을 클릭하면 App은 카테고리 선택 및 내용 입력창이 포함된 문의 작성 화면을 표시합니다. 사용자가 내용을 작성한 후 전송 버튼을 누르면 App은 필수 입력항목의 누락 여부를 먼저 검사합니다. 첫 번째 alt 분기문에서 입력값이 누락되었다면 서버 요청 없이 즉시 "모든 항목을 입력해 주세요"라는 팝업을 띄우고 흐름을 종료합니다. 반면 모든 항목이 정상적으로 입력되었다면 App은 Server에 Inquiry.submit 요청을 보냅니다.

요청을 받은 Server는 해당 문의 데이터를 DB에 대기 상태(status = "대기")로 저장합니다. 저장이 완료되면 두 번째 alt 분기문을 통해 전송 결과에 따른 처리가 이루어집니다. 전송 실패 시 Server는 오류 응답을 반환하고 App은 에러 팝업을 띄우며 기존 작성 내용을 그대로 유지합니다. 전송 성공 시에는 완결 응답을 받은 App이 사용자에게 "전송완료" 팝업을 보여준 뒤 메인 화면으로 이동합니다. 마지막으로 사용자가 작성 도중 취소 버튼을 누르거나 화면을 이탈하는 작성 중 취소(opt) 상황이 발생하면, App은 경고 알림창을 띄우고 사용자의 확인 선택 여부에 따라 메인 화면으로 이동하거나 기존 작성 화면을 유지하며 전체 프로세스를 마무리합니다.

### 3.10 Manage Inquiry
<img width="635" height="807" alt="image" src="https://github.com/user-attachments/assets/8e848960-35ae-4458-84ff-f0a29a80a722" />
<br/>

본 다이어그램은 관리자가 접수된 문의사항 목록을 조회하고 답변을 등록할 때 Admin, App, Server, DB 간에 이루어지는 데이터 흐름과 조건별 처리 과정을 나타냅니다. 전체 흐름은 대기 상태의 문의 조회, 특정 문의 상세 확인, 그리고 답변 저장 결과에 따른 화면 갱신 단계로 진행됩니다.

관리자가 관리자 페이지에서 '문의 관리' 메뉴를 클릭하면, App은 Server에 대기 상태인 문의 목록(Inquiry 목록)을 요청합니다. Server는 DB에서 status가 "대기"인 Inquiry 데이터를 조회하여 반환받으며, 이때 첫 번째 alt 분기문이 작동합니다. 접수된 문의가 없는 경우 Server는 빈 목록을 반환하고 App은 비어있는 화면을 표시합니다. 반면 문의가 존재하는 경우 대기 중인 문의 목록이 반환되어 App 화면에 출력됩니다.

관리자가 목록에서 특정 문의를 선택하면 App은 Server에 상세 조회를 요청하고, Server는 DB를 통해 해당 문의의 카테고리, 내용, 접수일 등 상세 데이터를 가져와 답변 입력란이 포함된 상세 화면을 App에 띄워줍니다. 이후 관리자가 답변을 작성하고 '답변 등록' 버튼을 누르면, App은 Server에 Inquiry.addAnswer 요청을 보냅니다. Server는 DB에 답변(answer)을 저장하고 해당 문의의 상태를 "완료"로 업데이트한 뒤, 두 번째 alt 분기문을 실행합니다. 저장 실패 시 오류 응답을 보내 작성 중이던 답변 내용을 유지시키고, 저장 성공 시에는 처리 완료 응답을 보냅니다. 마지막으로 이를 받은 App이 문의 상태를 "완료"로 화면을 갱신하면서 전체적인 관리 프로세스가 완료됩니다.

본 시스템의 전체 상태 머신 다이어그램은 사용자의 인증 세션 및 핵심 기능들의 생명주기를 단일 흐름으로 추적한다. 앱이 실행되면 사용자는 가장 먼저 비인증 상태(Unauthenticated)인 게스트 모드로 진입하게 되며, 이 상태에서 회원가입을 거쳐 최종적으로 로그인에 성공하면 사용자 권한을 확인한 뒤 인증 완료 상태(Authenticated)의 메인 대시보드로 이동한다. 대시보드가 로드되면 추천 클래스를 통해 인기 레시피 목록을 출력하고, 사용자가 특정 레시피를 선택함에 따라 조회수가 누적되면서 상세 조회(RecipeViewing) 상태로 전이된다. 상세조회 상태 내부에서는 북마크 추가나 삭제에 따라 스크랩 상태가 동적으로 토글되거나 식재료 및 도구의 가이드 팝업이 활성화되는 보조 액션들이 수행되며, 여기서 사용자가 변환 기능을 실행하면 레시피 변환(RecipeAdapting) 상태로 진입한다. 변환 상태에서는 어댑터 인스턴스가 동작하여 인분 수 변경에 따른 수량 비례 계산과 대체 재료 및 도구 선택 과정을 거쳐 최종적으로 커스텀된 새로운 레시피를 동적으로 생성해 낸다. 한편, 사용자가 1:1 문의 센터를 활성화하면 문의 처리(InquiryHandling) 상태로 전이되어 사용자가 제출한 문의 글이 데이터베이스에 대기(Pending) 상태로 등록되며, 이후 관리자 권한을 가진 계정이 답변을 등록하고 상태를 업데이트함에 따라 최종 완료(Completed) 상태로 전이되는 고객 지원 라이프사이클을 갖는다. 이러한 모든 시스템의 기능적 상태들은 사용자가 로그아웃을 수행하여 다시 비인증 상태로 복귀하거나, 애플리케이션 자체를 종료하여 최종 종료 상태(Final State)에 도달함으로써 유기적인 하나의 실행 주기를 마무리하게 된다.
